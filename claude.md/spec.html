1. Vision: From ESG Tool to Planetary Intelligence Terminal

GreenOrb must transcend its current positioning as a corporate ESG auditor. The goal is to become the single platform from which any stakeholder — investor, policymaker, regulator, journalist, or citizen — can look at planet Earth in real time and understand what is happening to it: what companies are doing, what the atmosphere is doing, what ecosystems are doing, and where the planet's vital signs are deteriorating or improving.

North Star Metric

Any user opening GreenOrb should be able to answer — within 60 seconds, without prior training — the question: 'What is the single most urgent environmental event happening on Earth right now, and which organisations are responsible for it?'

Achieving this requires three platform shifts happening simultaneously: (1) a globe engine upgrade that can handle hundreds of simultaneous high-density data layers without frame drops, (2) a complete data source expansion from the current 8 ESG RSS feeds to a 40+ source planetary monitoring network, and (3) a new intelligence fusion layer that correlates signals across domains — corporate, satellite, meteorological, ecological, and geopolitical — and surfaces the most important convergences automatically.

2. Globe Engine Upgrade: CesiumJS as the New Foundation

2.1 Why the Current Globe Falls Short

The current react-globe.gl / globe.gl stack is excellent for prototyping and handles 10,000–50,000 point primitives well. However it has hard architectural limits that will prevent GreenOrb from becoming the platform described in Section 1. The specific blockers are: no native terrain support (no ability to show topographic relief, elevation-based flooding zones, or mountain fire fronts), no time-dynamic tile streaming (no ability to play back historical satellite imagery day-by-day), no WMS/WMTS tile protocol support (blocks integration with NASA GIBS, Copernicus, NOAA, and dozens of other planetary data providers), and vertex count limits that prevent simultaneous rendering of species distribution polygons, ocean current vectors, fire perimeter polygons, and corporate facility markers at the same time.

2.2 CesiumJS: The Recommended Replacement

Recommendation

Replace globe.gl / Three.js with CesiumJS (open source, Apache 2.0). CesiumJS is the same engine used by NASA, the US Air Force, Google Earth Web, and Microsoft Flight Simulator for terrain. It handles 100M+ polygons, native 3D tiles, time-dynamic imagery, WMS/WMTS/TMS/CZML/KML/GeoJSON/MVT all natively.

2.3 Migration Path

The migration does not require rewriting all layers at once. The recommended path is a three-sprint phased migration:

▸ Sprint 1 — Install resium and CesiumJS alongside the existing globe. Migrate the Indore Smart City tab first (lowest data density, best for validation).

▸ Sprint 2 — Migrate the Globe Tab. Port existing pointsData, ringsData, polygonsData primitives to Cesium Entity/Primitive equivalents. The layer config system in layers.config.js remains unchanged — only the renderer changes.

▸ Sprint 3 — Enable the new capabilities that justify the migration: terrain, time-dynamic imagery, and WMS tile streaming. These are the layers described in Sections 4–10 of this document.

Key implementation note: all existing API endpoints, TTL/cache logic, and the LangGraph agent pipeline are completely unaffected by the globe engine change. The backend contract does not change.

3. Satellite Constellation Layers

3.1 Why Satellites Matter for ESG

The UCS Satellite Database (provided as source data for this document) contains 1,738 catalogued active satellites across 60+ countries. From an ESG and planetary monitoring perspective, satellites are not just orbital objects — they are the eyes, ears, and data pipelines of the planetary monitoring network. Every real-time data feed GreenOrb currently shows — FIRMS fire data, MODIS imagery, air quality, ocean colour — originates from one of these satellites. Showing the orbits of the very satellites generating your data is a powerful visual that makes the platform feel alive and scientific rather than just a dashboard of numbers.

Key Insight from Dataset

The provided CSV contains 1,738 rows covering satellites from 60+ operator countries. The dataset includes orbital parameters (perigee, apogee, inclination, period), launch dates, contractor countries, launch vehicles, and purpose classifications. All of this can be used to build filterable, searchable satellite layers with rich tooltip content.

3.2 Orbital Mechanics Implementation

Do not attempt to render satellite positions from a static lat/lng coordinate. Satellites move at 7.8 km/s in LEO — any static position is already thousands of kilometres wrong. The correct implementation uses Two-Line Element (TLE) sets and the SGP4 orbital propagation algorithm, executed client-side in a Web Worker:

▸ Data source: CelesTrak REST API (https://celestrak.org/SOCRATES/query.php) provides TLE sets updated every 2 hours for all catalogued objects.

▸ Propagation: The satellite.js npm library (MIT licensed) implements SGP4/SDP4 entirely in JavaScript. No server computation needed.

▸ Update cadence: Re-propagate positions every 3 seconds. At 3-second intervals, a LEO satellite moves ~23 km — visually acceptable for a globe visualization.

▸ Web Worker isolation: Run all SGP4 math in a dedicated Web Worker (public/workers/satellite-propagator.worker.js). The main thread receives lat/lng/alt arrays ready to render.

▸ Cesium rendering: Use Cesium.SampledPositionProperty with Cesium.CallbackProperty for smooth trail rendering. Cesium's built-in CZML format can encode entire satellite passes natively.

4. Natural Disasters — Real-Time Planetary Hazard Layer

4.1 Architecture Overview

Natural disaster data must be treated as a first-class, always-on layer — not an optional toggle. Every disaster event is simultaneously an environmental data point (emissions from fires, ecosystem damage from floods), a financial data point (supply chain disruption, insurance exposure), and an ESG data point (corporate facilities at risk, BRSR reporting obligations triggered). The disaster layer connects all three.

Data Philosophy

Do not rely on a single aggregated disaster API. Use primary government and scientific agency feeds directly. Each agency has different coverage, latency, and data quality. Redundancy ensures GreenOrb never shows 'no events' when events are happening.

DIS Implementation Note

DIS is computed in the convergence.worker.js Web Worker, not on the backend. This keeps latency low and avoids server load. The worker receives raw disaster events from the backend API, enriches them client-side with population data (cached in IndexedDB), and emits scored events to the globe renderer.

4.4 Disaster-to-Company Correlation Engine

This is GreenOrb's highest-value disaster feature and has no equivalent in any other ESG tool. The engine works as follows:

▸ Step 1: For every active disaster event, compute which tracked companies have facilities (lat/lng in the companies table) within configurable risk radii: 25km (direct), 100km (operational disruption), 500km (supply chain disruption).

▸ Step 2: Query the strategies table for those companies. If a company has a BUY or HOLD rating and is within 100km of a major disaster, downgrade the confidence score automatically and set a 'Physical Risk Override' flag.

▸ Step 3: Trigger a red ring pulse animation on the globe at the company's facility location. The ring's pulse frequency scales with the DIS severity score.

▸ Step 4: Generate an automated AI summary: 'Company X's [facility type] in [location] is within [N]km of an active [disaster type] (DIS: [score]). This may affect [operations/supply chain/regulatory reporting].'. Cache this summary in Redis with a 15-minute TTL.

▸ Step 5: Feed this event into the Trust UI queue as a new AI-generated finding for human auditor review.

5. Weather and Precipitation — Beyond Simple Overlays

5.1 Design Philosophy

Generic weather is not the goal. Every weather layer in GreenOrb must have a direct ESG interpretation. Rain is interesting to a farmer; rain that falls on a coal mine in Jharkhand and carries runoff into a river is interesting to an ESG analyst. The data is the same — the context transforms it into intelligence.

5.2 Precipitation: The GPM Constellation

The Global Precipitation Measurement (GPM) mission, a joint NASA/JAXA program, provides near-global precipitation estimates updated every 30 minutes with a 0.1-degree spatial resolution. This is the gold standard for real-time precipitation data and should be GreenOrb's primary weather data source.

5.3 Precipitation Intelligence — Moving Beyond 'How Much Rain'

Raw precipitation volume is a single number. GreenOrb should compute and display derived precipitation intelligence that tells users what the rain means for ESG outcomes:

▸ Drought Index Anomaly: Compare current 30-day precipitation against 30-year climatological normal for that grid cell (CHIRPS dataset). Display as a deviation map — cells more than 2 standard deviations below normal are flagged as drought-stressed.

▸ Flood Risk Accumulation: When a grid cell receives >150% of its monthly normal in <7 days, flag it as acute flood risk. Cross-reference against tracked facility locations.

▸ Agricultural Stress Index: Combine soil moisture (SMAP satellite), temperature (ERA5 reanalysis), and precipitation anomaly to produce a crop stress score per admin district. Used for food security and agricultural supply chain ESG context.

▸ Water Stress Amplifier: When precipitation falls on regions already classified as 'extremely high water stress' by the WRI Aqueduct database, the scarcity risk increases non-linearly. Visualise this as a pulsing overlay on the water stress layer.

5.5 Weather Display Implementation

For all continuous field layers (precipitation, temperature, wind), the recommended rendering approach is a CesiumJS WebGL fragment shader applied to a raster tile layer, not individual data points. This produces smooth, GPU-accelerated colour gradient maps at any zoom level with zero performance penalty. The implementation:

▸ Backend: Download NetCDF/GeoTIFF from source API on schedule. Convert to Cloud-Optimised GeoTIFF using GDAL. Cache in the backend's /api/weather/* endpoints with appropriate TTLs (30min for GPM, 6hr for ERA5).

▸ Frontend: CesiumJS ImageryLayer with a custom ColorMapMaterialProperty shader. Use a scientifically appropriate colormap: viridis for precipitation, RdBu for temperature anomaly, Spectral for vegetation.

▸ Tooltip: When user hovers over a precipitation area, show a tooltip: 'Precipitation: 42mm in last 24h (+180% vs. 30-year normal). River flood risk: HIGH'. Use the Gemini Flash API to generate a one-sentence contextual summary if the value is anomalous.

6. Biodiversity and Species Distribution

6.1 Why Biodiversity Belongs in GreenOrb

The Kunming-Montreal Global Biodiversity Framework (2022) has created mandatory nature-related disclosures for corporations, similar to how the Paris Agreement created carbon disclosures. SEBI's BRSR framework (Principle 6) explicitly requires disclosure of whether company operations are in or adjacent to biodiversity-sensitive areas. The TNFD (Taskforce on Nature-related Financial Disclosures) framework, which SEBI is expected to reference in future BRSR updates, requires companies to disclose nature-related risks. GreenOrb is uniquely positioned to automate all of this by overlaying species and habitat data directly against company facility locations.

Market Opportunity

No ESG platform currently provides automated biodiversity risk assessment against company facilities at the species level. TNFD compliance tooling is a €2B+ emerging market. GreenOrb can lead this space by being first to market with GBIF + facility lat/lng intersection at scale.

6.3 Biodiversity Globe Rendering Strategy

2.3 billion GBIF occurrence points cannot be rendered as individual markers. The rendering strategy must be hierarchical:

▸ Zoom level 0–3 (global): H3 resolution-2 hexagons (~86,000 km² per cell) coloured by species richness index. Computation: pre-aggregate from GBIF bulk download, store as H3 index in Postgres, serve via /api/biodiversity/richness endpoint.

▸ Zoom level 4–7 (regional): H3 resolution-5 hexagons (~252 km² per cell). Show individual IUCN threatened species occurrence density as a separate color channel.

▸ Zoom level 8+ (local/facility): Show individual species occurrence points with full GBIF metadata. Highlight all CR/EN species within the viewport.

▸ Facility overlay: At any zoom level, when a tracked company is visible, show a dashed radius ring at 50km around their facility coloured by the worst IUCN category of any species whose range intersects that circle.

7. Sustainability News Intelligence — The GDELT Integration

7.1 Current State vs. Required State

GreenOrb currently ingests 8 ESG RSS feeds. This is insufficient for three reasons: coverage (8 feeds misses 95%+ of sustainability reporting globally), latency (RSS feeds update every 15 minutes at best), and signal extraction (raw headlines provide no trend analysis, entity extraction, or velocity computation). The target state is a full GDELT integration that monitors 100,000+ global news sources in 65 languages in near real-time.

7.2 GDELT — The World's Largest Open Database of Human Society

9.3 Bootstrap Hydration — 52-Layer Edition

Extending the World Monitor bootstrap hydration pattern to 52 layers requires a three-tier parallel bootstrap instead of two:

▸ Ultra-fast tier (30-second cache): Disaster alerts (earthquake, fire, tsunami, cyclone), air quality. Fired immediately on page load. These are the 'is the world on fire right now?' queries.

▸ Fast tier (5-minute cache): Corporate data (companies, countries, trust alerts, news velocity), carbon grid, precipitation current. Fired 100ms after ultra-fast tier.

▸ Slow tier (1-hour to 24-hour cache): Satellite TLE sets, biodiversity richness, protected areas, land cover, water stress, forest alerts. Fired 500ms after fast tier. These are background-loaded and do not block initial render.

▸ Deferred tier (user-triggered): All Tier-3 compute-intensive layers. Never loaded on startup. Only fetched when user explicitly activates the layer.

9.4 Adaptive Layer Quality

On low-bandwidth connections (detected via Navigator.connection API), automatically downgrade rendering quality:

▸ Full quality: Individual TLE-propagated satellite positions, full-resolution precipitation rasters, H3 res-5 biodiversity hexagons

▸ Mid quality: Satellite constellation as constellation-average orbital ring (no individual positions), precipitation as country-level choropleth, biodiversity as H3 res-3 hexagons

▸ Low quality: No satellite layer, precipitation as single national average value, biodiversity as country-level flag only (in/out of KBA)

14. Data Ethics and Responsible Use Guidelines

GreenOrb's expanded data coverage raises important questions about responsible data use. As a platform claiming to expose greenwashing and corporate environmental harm, GreenOrb must hold itself to the same standard of evidence and epistemic honesty it demands of the companies it audits.

Core Principle

GreenOrb is an intelligence platform, not an enforcement platform. Every verdict, risk score, and convergence alert must carry an explicit confidence level, a traceable evidence chain, and a human-auditor override mechanism. The Trust UI is not optional — it is the epistemic backbone of the entire system.

▸ Accuracy-before-speed: When satellite data shows a potential emissions anomaly near a facility, the AI summary must state the detection method, the satellite and overpass time, the confidence of the attribution, and any known confounders (e.g., dust storm vs. industrial emission).

▸ Attribution uncertainty: GDELT articles may misattribute events to companies, especially in low-resource languages. All GDELT-derived findings must be flagged with 'requires human verification' until a Trust UI auditor confirms.

▸ Biodiversity data limitations: GBIF species occurrence data has massive geographic biases — well-sampled regions (Europe, North America) will show artificially high richness compared to under-sampled regions (Central Africa, Central Asia). All biodiversity metrics must display a sampling bias indicator.

▸ No weaponisation: GreenOrb data must not be used to enable discrimination against employees, to coordinate market manipulation (short-selling triggered by GreenOrb alerts), or to identify individuals based on GPS/location data. All spatial data is aggregated at minimum H3 resolution-6 before display.

▸ Right to respond: Any company with an active audit in GreenOrb has the right to submit a response to any AI-generated finding via the Trust UI. This response is displayed alongside the finding and incorporated into the final verdict.

15. Closing — The Platform Only GreenOrb Can Build

Every data source described in this document is free, open, or low-cost. CesiumJS is open source. GBIF is CC BY 4.0. GDELT is free. GPM IMERG is free. USGS earthquake data is free. The data infrastructure cost of building the platform described here is under $500/month in cloud compute and API costs.

What makes GreenOrb unique is not the data sources — it is the synthesis. No other product simultaneously ingests corporate ESG disclosures, satellite physical evidence, species range data, disaster event streams, precipitation anomalies, and news media velocity, and applies machine learning to find convergences across all of them. That synthesis — the automated cross-domain intelligence that asks 'what does the rain data, the satellite imagery, the news coverage, and the company's Scope 3 disclosure together tell us about what this company is actually doing to the planet?' — is a capability that currently requires a team of 20 analysts to produce manually.

GreenOrb automates that team. And it does so transparently, with every finding traceable to a primary source, every verdict subject to human override, and every data limitation disclosed to the user.
