✅ RECOMMENDATION: Do NOT Migrate to CesiumJS in Sprint 0 or Sprint 1

The disaster layers, satellite orbits, earthquake data, and GDELT news velocity that Sprint 0 & 1 target work perfectly well on react-globe.gl using the existing pointsData, ringsData, and polygonsData primitives.

Migrating the globe engine during Sprint 0 introduces a 3–5 day delay, a large bundle size penalty, and significant risk of breaking the Compare Tab, Trust UI, and Indore Smart City dashboard — all of which are currently working.

The correct time to migrate is Sprint 3, when precipitation rasters (GPM IMERG) and WMS tile streaming are needed. Those features genuinely require CesiumJS. Earthquake dots and satellite orbits do not.

Verdict: Decline the CesiumJS migration for now. Add satellite.js for SGP4 propagation. Keep react-globe.gl as the renderer. Revisit at Sprint 3.

TASK S0-2: Create satellite-propagator.worker.js

TASK S0-2 Create public/workers/satellite-propagator.worker.js

FILE: public/workers/satellite-propagator.worker.js ACTION: CREATE NEW FILE

WHY:

The Web Worker runs SGP4 propagation off the main thread so the globe UI never jitters during orbit calculations. It receives TLE data from the main thread, propagates all satellite positions every 3 seconds, and posts back an array of {lat, lng, alt, name, color} objects ready for pointsData.

IMPLEMENTATION:

→ Fetch TLE groups from CelesTrak on worker init: https://celestrak.org/SOCRATES/query.php?CATALOG=stations (ISS group), visual (Starlink), active (all active)

→ Parse each TLE pair with satellite.twoline2satrec(line1, line2)

→ Every 3000ms: call satellite.propagate(satrec, new Date()) for each satellite, convert ECI to geodetic with satellite.eciToGeodetic() + satellite.gstime()

→ Post message to main thread: { type: 'SAT_UPDATE', satellites: [{lat, lng, alt, name, color, category}] }

→ Error handling: if any satellite's position vector is false (below horizon / propagation error), skip it silently

→ TLE refresh: re-fetch from CelesTrak every 2 hours (TLEs have 2hr validity guarantee)

→ Start with 40 representative satellites (same set as the demo): ISS, Terra, Aqua, GOES-16, Sentinel-1A/2A/2B/3A, GPS IIF-10/12, GLONASS-M 755, Galileo FOC-7, BeiDou-3, 4× Starlink, 2× OneWeb, Worldview-3, Jason-3

→ Colour coding by category: nav=teal, weather-geo=amber, comms-mega=indigo, eo=blue, crewed=pink, climate=purple

Effort: 4 hours Risk level: LOW

TASK S0-5: Upgrade useGlobeData.js to 3-Tier Bootstrap Hydration

TASK S0-5 3-Tier Bootstrap Hydration in useGlobeData.js

FILE: src/hooks/useGlobeData.js ACTION: MODIFY

WHY:

Currently each globe layer fetches independently as it mounts. With 8+ layers this creates a waterfall of individual HTTP requests that delays first meaningful render. The bootstrap hydration pattern fires all fetches in parallel, in priority groups, from a single coordinated hook. The globe loads instantly because data arrives in a predictable order.

IMPLEMENTATION:

→ Tier 1 (fire immediately on mount, 30-second cache): /api/disasters/earthquakes, /api/globe/fires. These are the 'is anything on fire right now?' queries — always first.

→ Tier 2 (fire 100ms after Tier 1, 5-minute cache): /api/globe/companies, /api/globe/countries, /api/globe/air-quality, /api/globe/grid. Core ESG data.

→ Tier 3 (fire 500ms after Tier 2, 1-hour cache): /api/globe/news-velocity, /api/globe/assets (Climate TRACE). These are background enrichment — never block render.

→ Implementation: Promise.all() within each tier, sequential setTimeout(fn, delay) between tiers

→ Add stale-on-error: if any fetch throws, serve last cached value from a React ref (not setState — avoids re-render jitter)

→ Add hidden-tab throttle: document.addEventListener('visibilitychange', ...) — multiply all polling intervals by 10 when tab is hidden

→ Add 15% jitter to all intervals: interval * (0.85 + Math.random() * 0.3) — prevents synchronized API storms across multiple browser tabs

Effort: 4 hours Risk level: MEDIUM — test that each tier fires correctly and stale flag shows in UI

TASK S0-6: Harden nasa-firms.js with Negative Caching

TASK S0-6 Add negative caching and stale-on-error to Backend/api/nasa-firms.js

FILE: Backend/api/nasa-firms.js ACTION: MODIFY

WHY:

NASA FIRMS is an external API that goes down occasionally. When it does, the current code returns an empty array and the fires layer shows blank. The fix implements negative caching: when FIRMS returns an error, cache a sentinel value and serve the last good data with stale:true for up to 10 minutes. Users see data; they just see a small stale indicator.

IMPLEMENTATION:

→ Wrap the FIRMS fetch in try/catch. On success: cache the response with a 1h TTL (existing behaviour).

→ On failure: check if a last-good cache entry exists. If yes: return it with { data: lastGood, stale: true, cached_at: lastGoodTimestamp, source: 'NASA FIRMS (cached — source unavailable)', ttl: 600 }

→ If no last-good entry exists either: return { data: [], stale: true, source: 'NASA FIRMS (unavailable)', ttl: 60 } — empty is better than crashing

→ Use exponential backoff for retries on the backend: first retry at 60s, then 120s, then 300s. Stop retrying after 3 failures — wait for next scheduled poll.

→ This same pattern should be applied to all external API routes: grid-carbon.js, air-quality.js, climate-trace.js. The pattern is identical — wrap in try/catch, serve stale on error.

Effort: 2 hours Risk level: LOW

3.3 New Backend Disaster API Routes

3.3.1 Backend/api/floods.js

Source: GDACS (UN OCHA) RSS feed — free, no key, global coverage. URL: https://www.gdacs.org/xml/rss.xml. The GDACS feed covers multiple hazard types; filter to eventtype='FL' (flood). Parse the RSS XML with a lightweight parser (xml2js or fast-xml-parser — both already likely in node_modules). Extract: lat, lng, severity (Green/Orange/Red), country, eventname, date. Map severity to colour: Red=#ef4444, Orange=#f97316, Green=#eab308. TTL: 1800 seconds (30 minutes).

Note: GDACS also provides earthquakes, cyclones, and tsunamis. The same route file can be structured as a multi-hazard endpoint and filtered by type, reducing the number of external HTTP calls from 4 to 1.

3.3.2 Backend/api/cyclones.js

Source: NHC (National Hurricane Center) active storms GeoJSON — https://www.nhc.noaa.gov/gis/forecast/archive/ — free, no key. For non-Atlantic storms: JTWC data at https://www.metoc.navy.mil/jtwc/jtwc.html. Additionally use IBTrACS via GDACS for global coverage. The key field is the storm track (sequence of lat/lng points with forecast cone). Render as a custom layer using a polyline per storm track. TTL: 3600 seconds (1 hour — storm tracks update hourly).

3.3.3 Backend/api/volcanoes.js

Source: Smithsonian Global Volcanism Program — https://volcano.si.edu/gvp_currenteruptions.cfm. This is an HTML page, not a JSON API. Parse it with cheerio (HTML parser, likely already in package.json — if not, add it). Extract the table of current eruptions: volcano name, country, lat, lng, activity level, last updated. Colour code by activity: Erupting=#ef4444, Unrest=#f97316, Declining=#fbbf24. TTL: 21600 seconds (6 hours — Smithsonian updates weekly for most volcanoes). Important: Always return the stale-on-error fallback — Smithsonian's page is not a formal API and can be slow.

3.3.4 Backend/api/nasa-eonet.js (Bonus — Zero Extra Work)

NASA EONET (Earth Observatory Natural Event Tracker) is a free JSON API that already aggregates fires, storms, volcanoes, and ice events: https://eonet.gsfc.nasa.gov/api/v3/events?limit=50&status=open. No key required. This single endpoint can supplement all other disaster sources as a fallback and provides event title, category, and geometry. Already demonstrated in the proof-of-concept demo. Add as a supplementary route at /api/disasters/eonet — 30 minutes of work, high value.

function computeDIS(event, companies, populationIndex, previousPoll) {

const mag = normalise(event.magnitude, event.type); // 0–1

const pop = getPopulation100km(event.lat, event.lng, populationIndex); // 0–1

const prox = nearestCompanyScore(event, companies); // 0–1

const growth = growthScore(event, previousPoll); // 0–1

const media = event.mediaVelocity || 0; // 0–1

return (mag*0.35 + pop*0.25 + prox*0.20 + growth*0.15 + media*0.05) * 100;

}

3.5 Disaster-to-Company Proximity Engine

3.5.1 Backend: disasters-proximity.js

A new backend route /api/disasters/proximity runs on a schedule (every 15 minutes) to compute which companies are near active disasters. It does not replace client-side DIS computation — it persists proximity alerts to the database for the Trust UI queue.

▸ Input: fetch /api/disasters/* (all 5 hazard types), fetch /api/globe/companies

▸ For each disaster event with DIS > 50: compute Haversine distance to all company lat/lng pairs

▸ If distance < 100km: write a record to proximity_alerts table: { company_id, disaster_type, disaster_lat, disaster_lng, distance_km, dis_score, alert_time, status: 'open' }

▸ If distance < 25km: mark as severity: 'critical' — these trigger immediate Trust UI notifications

▸ Route returns the current open alert list: GET /api/disasters/proximity

▸ Idempotency: before inserting, check if an alert for the same (company_id, disaster_type, disaster_lat.toFixed(1), disaster_lng.toFixed(1)) already exists and is open — avoid duplicate alerts for the same ongoing event
